# Task Plan: task0001 — commit-docs.sh (flock-serialized doc commits)

## Goal

A deterministic script that commits workflow artifacts inside the integration
worktree under the same lock merge-task.sh uses, so doc commits and task
merges never race.

## Requirements

FR3, NFR2, NFR3, NFR4

## Scope

### Files to Create
- `em-workflow/scripts/commit-docs.sh` — serialized add + commit inside a given worktree
- `tests/test_commit_docs.py` — stdlib unittest coverage on fixture repositories

### Files to Modify
- (none)

## Design

Implements the "Contract: commit-docs.sh" pinned in IMPLEMENTATION.md.

Processing flow:
1. Validate arguments: worktree path exists and is a linked git worktree;
   commit message non-empty. Failure → argument-failure exit code, no git
   state touched.
2. Resolve the shared lock file path from the worktree's git common
   directory (same file merge-task.sh locks: `em-workflow-merge.lock`).
3. Acquire exclusive flock (blocking). Failure to open/acquire → lock-failure
   exit code.
4. Stage all changes within the worktree; if nothing is staged, exit 0
   without creating a commit (no-op success).
5. Commit with the given message (pre-commit hooks run normally — no bypass).
   Git failure → git-failure exit code; the ref must not be advanced on any
   failure path.

Style/structure follows merge-task.sh (existing pattern: `die` helper,
fd-based flock, `set -euo pipefail` equivalents as used there).

## Acceptance Criteria (MANDATORY)

- [ ] AC-1: With pending changes in a fixture integration worktree, the
  script exits 0 and the integration ref advances by exactly one commit
  whose message equals the given argument.
- [ ] AC-2: With no pending changes, the script exits 0 and the ref is
  unchanged (no empty commit).
- [ ] AC-3: With a missing or non-worktree path, or an empty message, the
  script exits non-zero with the argument-failure code and creates no commit.
- [ ] AC-4: While another process holds the shared lock, the script blocks
  until release and then commits; a concurrent merge-task.sh-style ref
  advance and a commit-docs.sh commit both survive (no lost update), with
  distinct exit codes distinguishing lock failure from git failure.

## Test Notes

- AC-1..AC-3: unit tests on temporary fixture repos (create repo, integration
  branch, linked worktree; run script via subprocess).
- AC-4: integration-style test spawning a background lock-holder process on
  the same lock file, asserting blocking behavior and final history; mirrors
  the concurrency tests in tests/test_merge_task_journal.py.
- flock absence is out of test scope (environment prerequisite, as for
  merge-task.sh).

## Out of Scope

- Modifying merge-task.sh (its semantics are unchanged).
- Calling the script from protocol documents (task0002/0003/0004 own the
  documents that reference it).
