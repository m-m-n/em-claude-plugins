# Task Plan: task0007 — protocol contracts: lock-aware fix commits, wake-phase refresh, bootstrap resume

## Goal

Every integration-ref advance in the protocol documents holds the shared
lock, wake-phase state edits happen on a refreshed worktree with a defined
exit-4 recovery loop, and a branch without workflow.yaml has a deterministic
resume route.

## Requirements

FR4, FR5, FR6, NFR2, NFR3

### Files to Modify
- `em-workflow/references/review-phase.md` — lock-aware auto-fix commits
- `em-workflow/references/implement-phase.md` — wake-phase refresh-before-edit + exit-4 recovery
- `em-workflow/skills/develop/SKILL.md` — bootstrap lifecycle state + exit-4 recovery + project_root variable capture

## Design

Rework of review round 1 residual findings (reviews/round1.yaml):
2be071cdd65fe558, 454c5eba23d061ce, 16702654a8c83434 (caller side),
32cc5c216645a7fb, 93f4c66b52b3ebf2. Their suggestions are authoritative;
key decisions:

1. **review-phase.md (R4 commit)**: the develop-driven fix commit no longer
   runs a bare add+commit; it acquires the same shared lock
   (`$GIT_COMMON_DIR/em-workflow-merge.lock`) across staging of the
   authorized files and the commit (flock wrapper — commit-docs.sh cannot be
   reused as-is because fix commits stage arbitrary authorized source files).
2. **implement-phase.md (wake phase)**: after any reconcile that observed a
   ref advance, refresh the integration worktree FIRST, then read/edit
   workflow.yaml, then commit. Define the bounded exit-4 recovery loop for
   every commit-docs.sh call site: refresh to branch tip → re-apply the
   intended state transition → retry once; a second exit 4 stops with a
   report.
3. **SKILL.md Step A (bootstrap states)**: discovery distinguishes
   (a) branch + workflow.yaml → resume Step B; (b) branch WITHOUT
   workflow.yaml → resume the incomplete create-spec deterministically
   (re-enter create-spec against the existing branch/worktree); zero
   branches → new feature. Also state the same exit-4 recovery contract for
   Step B doc commits, and capture the project root into a shell variable
   (`git rev-parse --show-toplevel`) referenced as a variable in the
   worktree-add template instead of literal interpolation (matching
   requirements-spec-creator Phase 3's safe pattern).

## Acceptance Criteria (MANDATORY)
