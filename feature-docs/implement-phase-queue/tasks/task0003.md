# Task Plan: task0003 — PreToolUse launch guard

## Goal

A PreToolUse hook on `Task` calls denies double-launching an in-flight
implementer task and records every allowed implementer launch as a
`launched` journal event, keeping the journal fully machine-written.

## Requirements

FR2, FR5, NFR1, NFR2, NFR4

## Scope

### Files to Create
- `em-workflow/hooks/queue_launch_guard.py` — the PreToolUse hook
  (self-contained, stdlib only)
- `tests/test_queue_launch_guard.py` — unittest coverage
  (subprocess-driven, JSON on stdin per `test/README.md`)

### Files to Modify
- (none — hooks.json registration is owned by task0005)

## Design

Follow IMPLEMENTATION.md: Journal contract (event schema, state derivation,
flock append), Task-identity discovery contract, fail-open convention, and
design decision D2.

Behavioral flow on each PreToolUse event for the `Task` tool (stdin =
PreToolUse JSON with `tool_input`):

1. Identify the call per the Task-identity discovery contract (subagent
   type `em-workflow:implementer` when available, else the
   `# Task assignment` block). Not an implementer launch → exit 0 with no
   output (no decision; normal flow applies).
2. Extract and validate `task_id` and `worktree_path`; derive the journal
   path. Validation failure → exit 0 (fail-open: the orchestrator's own
   fail-closed identifier gate governs real launches).
3. Replay the journal for that task id:
   - Last event `launched` (in-flight) → DENY with an explanatory reason
     naming the task id (the PreToolUse deny output contract, as
     bash_guard.py uses).
   - Last event `merged` → DENY (relaunching a merged task is always a
     mistake) with a distinct reason.
   - No events, or last event `failed` (retry path) → allow: append a
     `launched` event under flock, then exit with no decision output so
     the normal permission flow proceeds.
4. Journal unreadable/absent directory, malformed stdin → exit 0 silently.

Note on ordering: the `launched` append happens at allow time, before the
subagent actually starts (D2 stale-launch consequence is accepted and
bounded elsewhere; do not try to compensate in this hook).

## Acceptance Criteria (MANDATORY)

- [ ] AC-1: A first launch for a task (no journal events) is allowed and
  appends exactly one well-formed `launched` line (task id + RFC 3339
  `at`).
- [ ] AC-2: A launch for a task whose last event is `launched` is denied,
  with the task id in the deny reason; nothing is appended.
- [ ] AC-3: A launch for a task whose last event is `failed` (retry) is
  allowed and appends a new `launched` line.
- [ ] AC-4: A launch for a task whose last event is `merged` is denied;
  nothing is appended.
- [ ] AC-5: Task calls that are not em-workflow implementer launches
  (different subagent type / no Task-assignment block) produce exit 0 with
  no decision and no journal write.
- [ ] AC-6: Malformed stdin, invalid task id or worktree path, or an
  uncreatable journal location → exit 0, no crash.
- [ ] AC-7: The script imports only Python stdlib modules.

## Test Notes

- Subprocess tests with crafted PreToolUse JSON payloads (mirroring the
  implement-phase.md Task prompt payload) against a temp journal directory.
- Assert deny decisions via the hook's stdout/exit-code contract exactly as
  Claude Code consumes it (same style as would apply to bash_guard.py).
- Cover the malformed-journal-line skip (a bad line between valid ones must
  not flip the derivation).

## Out of Scope

- hooks.json registration (task0005).
- Blocking turn-end or recording failures (sibling hooks).
- Slot-count enforcement (the cap is the Stop hook's and the protocol's
  concern; this guard is strictly per-task identity).
