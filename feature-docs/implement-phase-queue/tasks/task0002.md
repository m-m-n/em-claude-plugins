# Task Plan: task0002 — Stop hook (queue loop guard)

## Goal

A Stop hook deterministically blocks the orchestrator from ending its turn
while refillable implementer slots exist, turning forgotten refills into
caught failures instead of silent stalls.

## Requirements

FR2, FR4, NFR1, NFR4

## Scope

### Files to Create
- `em-workflow/hooks/queue_stop_guard.py` — the Stop hook (self-contained,
  stdlib only)
- `tests/test_queue_stop_guard.py` — unittest coverage (subprocess-driven,
  JSON on stdin per `test/README.md`)

### Files to Modify
- (none — hooks.json registration is owned by task0005)

## Design

Follow IMPLEMENTATION.md: Journal contract (state derivation rule),
fail-open convention, line-based workflow.yaml reading, the
`MAX_PARALLEL_IMPLEMENTERS = 6` constant, and the Stop-guard sidecar
contract.

Behavioral flow on each Stop event (stdin = Claude Code Stop-hook JSON):

1. Resolve the project root (current working directory is the session's
   project directory for Stop hooks; a git-toplevel probe may refine it).
   Locate features via `feature-docs/*/workflow.yaml`; select those whose
   `implement` step is `in_progress` (line-based detection). None → exit 0.
2. For the active feature, read the task-id set from workflow.yaml (line
   patterns; ids validated per the contract) and replay the journal to
   classify each task: unlaunched / in-flight / merged / failed.
3. Decision:
   - Any task failed (last event `failed`) → exit 0 (user decision
     pending; never block).
   - No unlaunched tasks, or no free slot (in-flight ≥ 6) → exit 0.
   - Otherwise → BLOCK: exit 2 with a stderr message naming the feature,
     the free-slot count, and the specific task ids to launch next
     (ascending id order, at most free-slot count of them).
4. Loop cap via the sidecar: the derived state fingerprint (sets of
   unlaunched + in-flight task ids) is compared with the sidecar's last
   fingerprint. Same fingerprint → increment counter; different → reset to
   1. If the counter would exceed 3, do NOT block: emit a warning on
   stderr and exit 0 (the user takes over). The sidecar records fingerprint
   and counter; a successful non-blocking pass clears it.
5. Any unexpected condition anywhere (unreadable/malformed files, stdin not
   JSON, validation failures) → exit 0 silently (fail-open).

Also honor the Stop-hook re-entry field in the hook input (the flag Claude
Code sets when the session is already continuing due to a Stop hook) if
present: treat it as one more blocked-state signal feeding the same cap
logic, never as an unconditional block.

Multiple simultaneously active features: block for the first (stable
ordering by feature name) that has refillable work — one actionable
instruction per stop is enough; the hook fires again on the next stop.

## Acceptance Criteria (MANDATORY)

- [ ] AC-1: implement in_progress + free slots + unlaunched tasks → exit 2,
  and the stderr message names the exact task ids to launch (bounded by
  free slots, ascending order).
- [ ] AC-2: A task whose last journal event is `failed` → exit 0 (no block)
  even when unlaunched tasks and free slots exist.
- [ ] AC-3: In-flight count at 6 (no free slot) → exit 0; zero unlaunched
  tasks → exit 0; no feature with implement in_progress → exit 0.
- [ ] AC-4: Three consecutive blocks with the same state fingerprint, then
  a fourth stop in that state → exit 0 with a warning on stderr; a state
  change in between resets the cap.
- [ ] AC-5: Missing journal (phase just started), malformed journal lines,
  malformed stdin, or missing feature-docs → exit 0, no crash, no output
  requirement.
- [ ] AC-6: A `failed` task followed by a later `launched` (retry) counts
  as in-flight again — it neither suppresses blocking as "failed" nor
  counts as unlaunched.
- [ ] AC-7: The script imports only Python stdlib modules.

## Test Notes

- Tests build a temporary project directory (feature-docs +
  `.claude/worktrees/...` journal) and invoke the hook as a subprocess with
  crafted stdin JSON, asserting on exit code and stderr content.
- AC-4 drives the sidecar across sequential invocations in one temp dir.
- Edge cases from SPEC.md: exactly-6 in flight, zero pending, malformed
  journal line skipped, absent worktree directory.

## Out of Scope

- hooks.json registration (task0005).
- Launch-denying or failure-recording (sibling hooks).
- Any timeout/hang detection (watchdog is out of feature scope).
- Writing workflow.yaml or the journal (this hook only reads them; its only
  write target is its own sidecar).
