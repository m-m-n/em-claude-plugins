# Task Plan: task0001 — merge-task.sh journal append

## Goal

`merge-task.sh` records a `merged` event in the feature journal on every
successful merge, giving the queue loop its authoritative completion signal.

## Requirements

FR2, FR3, NFR2, NFR4

## Scope

### Files to Create
- `tests/test_merge_task_journal.py` — unittest coverage for the journal
  append behavior (throwaway git repos per `test/README.md`)

### Files to Modify
- `em-workflow/scripts/merge-task.sh` — append the `merged` event on
  success paths

## Design

Follow the Journal contract in IMPLEMENTATION.md exactly (path formula,
event schema, flock-on-journal append discipline, RFC 3339 timestamps).

- The journal path is derived from the worktree the script runs in: the
  task worktree lives at `.../em-workflow/{feature}/{taskNNNN}`, so the
  journal is the sibling `journal.jsonl` in the parent directory. Derive it
  from the worktree's own location (top level of the current work tree),
  validating the feature and task-id path segments per the contract before
  use. When the script runs outside that layout (e.g. tests exercising only
  merge behavior, or manual invocation), journal append is skipped
  silently — merging must not break where no journal home exists.
- Append on ALL exit-0 paths: fast-forward merge, true merge-commit, and
  the idempotent "already contained in parent" path (design decision D4).
  The `commit` field carries the parent-branch head that now contains the
  task (for the idempotent path: the current parent head).
- No append on conflict (exit 1) or error (exit 2) paths.
- The append itself must not turn a successful merge into a failure: if the
  journal write fails, report a warning on stderr but keep exit 0 (the
  merge DID happen; the orchestrator's git-state reconcile covers the gap).
- One line, single write, under an exclusive flock on the journal file
  itself (a separate lock from the merge lock; nested acquisition is fine).

## Acceptance Criteria (MANDATORY)

- [ ] AC-1: A fast-forward merge appends exactly one well-formed `merged`
  JSONL line with `task`, `commit`, and RFC 3339 `at` fields.
- [ ] AC-2: A merge-commit (non-fast-forward) merge appends exactly one
  well-formed `merged` line.
- [ ] AC-3: The idempotent already-merged path appends a `merged` line, and
  a resulting duplicate `merged` for the same task is well-formed.
- [ ] AC-4: A conflicting merge (exit 1) and a precondition failure
  (exit 2) append nothing.
- [ ] AC-5: When the worktree layout provides no journal home, the merge
  succeeds with no append and no error.
- [ ] AC-6: A journal write failure (e.g. unwritable directory) does not
  change the merge's exit code 0.
- [ ] AC-7: Concurrent merges from parallel worktrees produce no torn or
  interleaved journal lines (every line parses as JSON).

## Test Notes

- All ACs are integration-style tests against throwaway git repositories
  built in a temporary directory: create a base repo, an integration
  branch, task worktrees matching the `.claude/worktrees/em-workflow/...`
  layout, commits, then invoke the script as a subprocess.
- AC-7 launches several concurrent subprocess merges (distinct task
  branches) and asserts every journal line is parseable JSON with the
  expected task ids.
- Timestamp assertion: parseable as RFC 3339 with an offset, not an exact
  value.

## Out of Scope

- Any change to merge semantics, exit codes, or the conflict protocol.
- Writing `launched` / `failed` events (hooks own those).
- workflow.yaml updates of any kind.
