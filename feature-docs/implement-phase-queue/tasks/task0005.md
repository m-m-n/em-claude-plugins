## Design

Follow IMPLEMENTATION.md (Journal contract, `MAX_PARALLEL_IMPLEMENTERS = 6`
constant, design decisions D1–D3) and SPEC.md FR1 for the loop semantics.

### implement-phase.md Step I.2 rewrite

Replace the chunked model ("split into sequential chunks", "launch the
chunk's implementers in a SINGLE message", per-chunk barrier and cleanup)
with the queue loop:

1. Launch phase: pick unlaunched tasks (ascending id) up to 6 minus the
   current in-flight count, create their worktrees (existing I.2.a
   mechanics and resume guard unchanged), launch each implementer as a
   BACKGROUND `Task()` (synchronous fan-out explicitly forbidden — it
   reintroduces the barrier), then END THE TURN.
2. Wake phase (on completion notification): reconcile — replay the journal
   (last-event rule) and cross-check git state (worktree existence,
   `merge-base --is-ancestor` trust-but-verify, exactly as today) — update
   workflow.yaml task statuses, clean up merged tasks' worktrees/branches
   and refresh the integration worktree (existing I.2.d mechanics), then
   refill free slots and end the turn again.
3. Failed handling: on any failed task, stop new launches, let in-flight
   tasks drain, then the existing three-option AskUserQuestion (retry /
   route back to planning / abort — no skip), unchanged from current
   I.2.c.
4. Completion: all tasks merged → existing Step I.3 unchanged.
5. Document the supporting cast: journal contract summary (with a pointer
   that the raw log is machine-written and never deleted), the three hooks
   and what each catches, the stale-`launched` caveat and its bounds
   (Stop-hook cap + git reconcile), and the resume story (workflow.yaml +
   journal + git state; existing I.2.a guard retained).

Keep every part of the document this feature does not change: Step I.0
preconditions (including the identifier gate and environment probes),
Step I.1, prompt payload, command-approval flow, failure containment.

### hooks.json

Add three entries following the existing `${CLAUDE_PLUGIN_ROOT}` pattern
with sensible timeouts: Stop → `queue_stop_guard.py`; PreToolUse with a
`Task` matcher → `queue_launch_guard.py`; SubagentStop →
`queue_failure_net.py`. The existing PreToolUse Bash entry stays intact.
(The hook scripts are delivered by sibling tasks; this task owns the
wiring — integration-owner rule. The registration test asserts existence,
which holds once the feature's branches merge.)

### Documentation role split (README, workflow-schema)

One consistent statement in both places: `journal.jsonl` = machine-written
raw event log (append-only, never deleted, post-mortem primary source);
`workflow.yaml` = LLM-managed summary and SSOT; scripts/hooks never write
workflow.yaml. workflow-schema.md's sibling-artifacts section additionally
names the journal's path formula (outside feature-docs, under
`.claude/worktrees/`).
