# Task Plan: task0004 — SubagentStop failure net

## Goal

A SubagentStop hook records a `failed` journal event whenever an em-workflow
implementer stops without a `merged` event, so swallowed failures become
visible to the queue loop.

## Requirements

FR2, FR6, NFR1, NFR2, NFR4

## Scope

### Files to Create
- `em-workflow/hooks/queue_failure_net.py` — the SubagentStop hook
  (self-contained, stdlib only)
- `tests/test_queue_failure_net.py` — unittest coverage
  (subprocess-driven, JSON on stdin per `test/README.md`)

### Files to Modify
- (none — hooks.json registration is owned by task0005)

## Design

Follow IMPLEMENTATION.md: Journal contract, Task-identity discovery
contract, fail-open convention.

Behavioral flow on each SubagentStop event (stdin = SubagentStop JSON; the
hook fires for EVERY subagent stop, so filtering is the first concern):

1. Identify the stopped agent per the Task-identity discovery contract.
   The SubagentStop input's agent-type field is preferred when present;
   otherwise scan the agent's initial prompt / first user message (via
   whatever the input provides — inline prompt field or the agent
   transcript path) for the `# Task assignment` block. Not an implementer
   → exit 0 silently.
2. Extract and validate `task_id` and `worktree_path`; derive the journal
   path. Failure → exit 0.
3. Replay the journal for the task:
   - Last event `merged` → exit 0 (normal completion).
   - Last event `failed` → exit 0 (already recorded; do not duplicate).
   - Otherwise (`launched` or no event) → append a `failed` event under
     flock, with a one-line `reason` (e.g. "implementer stopped without a
     merged event").
4. Any unexpected input shape or unreadable state → exit 0 silently.

The hook never blocks the stop (it always exits 0 / non-blocking) — its
only job is the journal record; acting on the failure (halt launches, ask
the user) is the orchestrator's protocol, prompted via the Stop hook's
failed-task pass-through.

## Acceptance Criteria (MANDATORY)

- [ ] AC-1: An implementer stop whose task's last event is `launched`
  appends exactly one well-formed `failed` line (task id, `reason`,
  RFC 3339 `at`).
- [ ] AC-2: An implementer stop whose task's last event is `merged`
  appends nothing.
- [ ] AC-3: An implementer stop whose task's last event is already
  `failed` appends nothing (no duplicates).
- [ ] AC-4: Stops of non-implementer subagents (different type, no
  Task-assignment block) append nothing and exit 0.
- [ ] AC-5: Malformed stdin, missing transcript/prompt source, invalid
  identifiers, or an absent journal directory → exit 0, no crash.
- [ ] AC-6: The hook's exit is always non-blocking (exit 0) in every
  tested path.
- [ ] AC-7: The script imports only Python stdlib modules.

## Test Notes

- Subprocess tests with crafted SubagentStop JSON: cover both identity
  sources (agent-type field present; only a transcript file containing the
  Task-assignment block).
- Transcript-based tests write a minimal JSONL transcript fixture in a temp
  dir and point the input's transcript-path field at it.
- Cover the malformed-journal-line skip during replay.

## Out of Scope

- hooks.json registration (task0005).
- Deciding or executing the failure response (retry / route back / abort is
  the orchestrator's protocol).
- Hang detection — a subagent that never stops never fires this hook
  (watchdog is out of feature scope).
